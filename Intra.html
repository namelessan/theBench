Team agreement




Notions: teamwork








Role of the team agreement




Aaaaaand... it's back! The discuss-first-code-later part of a group project.
In case you forgot, here's why we write team agreements: writing a team agreement at the beginning of your group project clarifies expectations so that everybody knows what's their role and what part they should play in the project. Since everybody writes the team agreement together, it's easier to enforce its content: everybody already agreed to follow the rules, so it's less awkward to remind your teammates of what they should do if any conflict arises.
Note: the team agreement won't be formally graded, but if you don't do it seriously you will lose teamwork skill points. So. Do it seriously!












The process




Meet up with your teammates, read carefully the subject for the project and, before starting to code anything, create a file called team_agreement.md with the agreed on rules. Once it's finalised, push it!












What goes in it?




The team agreement should address the following points:
    

  • the name of the team
    • 

  • the working hours each member commits to
    • 

  • how often you will check-in with each others' progress
    • 

  • the strengths and weaknesses of each member
    • 

  • the allocation of the workload
    • 

  • your strategy so that you all understand the codebase
    • 

For the review, everybody needs to understand the key concepts of the project. So if you handled the queries but didn't look at the tools used for benchmarking, take a couple of hours to discuss them with your teammate; and vice versa if you only handled the benchmarking.














Benchmarking









Benchmarking




Notions: benchmarking








Introduction




As a developer, coding a functional program is not enough; you must ensure that your program consumes as little resources as possible - because servers cost money! 
To do so, you need to find ways to accurately measure performance: that's where analysis tools and benchmarking programs come in. Those pieces of software will help you produce a full report on which you can base yourself to improve your source code and enhance the performance of your program.












Your first mission




Write a program called benchmarking.py to calculate the memory allocation, run-time performance and number of function calls of the program passed as argument. 
You will implement the following options:
    

  •  -m outputs the memory allocation of the target program
    • 

  •  -t outputs the execution time (run-time) of the target program
    • 

  •  -n outputs the number of function calls of the target program
    • 

By default, the benchmarking program will run with -t if no option is passed.
Usage: ./benchmarking.py [-m] [-t] [-n] program

Libraries you may use




To launch a program within benchmarking.py you can use the run() function of the subprocess module.
1. To calculate memory allocation & run-time performance of the target program, you can use the resource module. (Note: it's the children you are tracking, not the benchmarking program itself.) If you want to see the memory usage at the function level, use tracemalloc.
2. To calculate the number of function calls, time per call... use profile or cProfile with the pstats library.












Your second mission




Here's a small exercice to use your program on. The exercice has to be pushed on the repository with the benchmarking program.
You will write a script called smart_db.py. That script will read on stdin a list of people. Each line will be of the format:
first_name,last_name,username,age,gender,city
Input acquisition stops at the first empty line.
The smart_db.py script will also take as argument a file with the query to run on the content read on stdin. You are free to use a JSON library to read the query file.
The query specifications are relatively simple:
    

  • The query will necessarily have a field "select" which indicates which fields should be printed out
    • 

  • The query may have a field "where_and": all conditions indicated in that field must be verified for a line to be returned
    • 

  • The query may have a field "where_or": at least one of the conditions indicated in that field must be verified for a line to be returned
    • 

  • The query will never have a field "where_and" and a field "where_or" at the same time. And there won't be anything nested.
    • 

  • The query may have a field "order" which indicates the field the result lines will be ordered from
    • 

  • The query field "left" will only contain a field name or the keyword first_letter + a field name
    • 

  • The query field "right" will only contain acceptable values for the given field name
    • 

  • The query field "op" will take one of those values: '<', '>', '=', '!='
    • 

That's all! Getting the answers from the queries shouldn't be too hard. However by benchmarking your exercice against different queries you will quickly realise that the way you store information affects the run-time of the different queries. And if you choose to optimise the run-time by storing more meta-information for easy retrieval... then you might become less memory-efficient!
What tradeoffs can you make? Can you optimise one performance without compromising the other?

Test files




We provide you below with one file for testing really large inputs.
You have to create, and push on your repository, at least 5 different queries to show that you tested your program under different constraints. (Let's say it's your third mission!)
We have noticed that because of the Sentinel you guys got quite lazy with making your own tests! Well, today let's change that. Try to think of the cases that might produce different performances (for example, if your list is ordered by first_name but your query requests an age > 30, the execution time is probably different than if your list was already ordered by age...) and benchmark your smart_db.py in those different conditions.

The review




For the review with mentors, you must have on your git repository:
    

  • The benchmarking.py program
    • 

  • The smart_db.py script
    • 

  • At least 5 example queries that show how your program behaves in different situations
    • 

Your benchmarking program must respect the following rules:
    

  • it doesn't change the content of the program
    • 

  • it doesn't use any external benchmarking libraries (to be clear: those are BANNED)
    • 

You will have to demo your project and explain your choices. What information did the initial benchmarking provide? How did you take it into account to improve performance? Are there critical cases you could not resolve? Prepare a presentation to show what you have done, step by step.
